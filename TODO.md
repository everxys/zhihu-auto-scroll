# Zhihu Auto Scroll 重构 TODO

审查基线：`zhihu-auto-scroll.js` v0.5.2。优先保持现有功能和默认行为不变，再改善可靠性、性能与可维护性。

## P0：先解决运行正确性与持续开销

- [x] **统一运行调度与生命周期，确保“暂停”会真正停止后台工作**
  - 现状：`pause()` 只切换状态并清理 `mutationRunTimer`；主 `setInterval`、`MutationObserver` 仍常驻，暂停后仍会持续唤醒脚本。暂停状态修改执行间隔还会重启主定时器。
  - 重构：建立单一运行控制器，集中管理 `start / pause / destroy`；暂停时清理主调度器、计时器和 MutationObserver，开始时再按需创建；在 `pagehide` 时执行完整清理。
  - 验收：暂停后不存在主循环定时任务和 DOM 观察器；重复开始/暂停不会产生多个定时器或观察器。

- [x] **为异步滚动流程增加可取消机制**
  - 现状：用户在 `runOnce()`、`scrollPage()` 或 `bounceAtBottom()` 的 `await sleep(...)` 期间点击暂停后，当前流程仍可能继续滚动、更新空闲轮数和面板状态。
  - 重构：使用运行代次令牌或 `AbortController`；每次异步等待和滚动前后校验当前任务仍有效；新一轮运行或暂停时取消旧任务。
  - 验收：在普通滚动和底部回弹的任意阶段点击暂停，后续不再发生滚动、点击或状态覆盖。

- [x] **用单一、自调度循环替代 `setInterval(async ...)` 与 MutationObserver 双重触发**
  - 现状：`setInterval(runOnce, ...)` 不等待异步任务结束，只依赖 `runBusy` 丢弃重叠调用；`requestRunSoon()` 又先调用 `expandAnswers()` 再调用会重复展开的 `runOnce()`。
  - 重构：使用一次性 `setTimeout` 在上一轮完成后安排下一轮；MutationObserver 只请求提前执行同一个调度器，不直接执行展开逻辑。
  - 验收：任意时刻最多存在一个主任务和一个待执行调度；一次页面变化最多触发一轮展开扫描。

- [x] **将全页 MutationObserver 和全量 DOM 扫描改为增量处理**
  - 现状：Observer 监听整个 `document.body`；每次有效变化最终都会扫描全页所有 `button, a, [role="button"]`，页面回答增多后成本持续上升；面板自身更新也可能触发观察回调。
  - 重构：忽略控制面板内部 mutation；优先观察回答列表容器；从 `addedNodes` 收集新增回答/按钮并批量处理，首次启动时才进行一次全量扫描。
  - 验收：更新计时器或面板状态不会安排主任务；新增一个回答时只检查新增子树，不重新扫描整页。

- [x] **收紧自动点击范围，避免误点页面上的同名控件**
  - 现状：脚本在整个页面仅凭文本匹配按钮，`显示全部`、`展开全部`、`继续阅读` 等文本并非回答区域专属；点击前就将元素放入 `WeakSet`，失败或无效果后不会重试。
  - 重构：把“展开当前回答”和“加载更多回答”拆成两类目标，分别限定在可靠的回答/列表容器内；检查 `disabled`、`aria-disabled` 等状态；使用带冷却和最大次数的尝试记录，并在确认点击有效后标记完成。
  - 验收：页头、侧栏、评论区和控制面板中的同名按钮不会被点击；临时失败的展开按钮仍可有限重试。

## P1：改善结构、兼容性与可验证性

- [x] **拆分单文件中的职责边界，但保留用户脚本的单文件交付**
  - 现状：配置、运行状态、滚动策略、DOM 匹配、持久化和面板渲染都共享同一批全局变量，修改一处容易影响其他流程。
  - 重构：在 IIFE 内按 `settings/storage`、`expander`、`scroller`、`scheduler`、`panel`、`app lifecycle` 分组；由 app 层协调模块，避免模块直接修改彼此内部状态。
  - 验收：每个模块有明确输入、输出和清理方法；滚动与展开逻辑不直接操作面板 DOM。

- [x] **为知乎 SPA 导航增加页面生命周期处理**
  - 现状：知乎站内切换问题时用户脚本可能不会重新注入，旧问题的运行状态、空闲计数和异步任务会延续到新页面。
  - 重构：检测问题 URL/问题 ID 变化；切换时取消当前任务、重置页面级状态，并按约定保持暂停或重新初始化；离开 question 页面时销毁运行资源。
  - 验收：站内连续切换多个问题不会残留旧任务、重复面板或沿用旧页面的到底判断。

- [x] **封装持久化访问并处理不可用、损坏和版本升级**
  - 现状：模块初始化和设置保存时直接访问 `localStorage`，存储被禁用或抛出异常会中断脚本；三个独立 key 没有配置版本。
  - 重构：增加 storage adapter，集中进行 `try/catch`、默认值、校验与版本迁移；评估改用油猴 `GM_getValue / GM_setValue`，避免配置暴露在页面的 localStorage 中。
  - 验收：存储不可用或内容损坏时脚本仍能使用默认配置运行；旧配置可平滑迁移。

- [x] **减少强制布局计算与重复 DOM 查询**
  - 现状：候选元素检查会对大量节点调用 `getComputedStyle()` 和 `getBoundingClientRect()`；`updatePanel()` 和计时器更新反复查询固定面板节点。
  - 重构：先做低成本的容器、文本和状态过滤，再做可见性检查；缓存面板节点引用，只渲染发生变化的字段。
  - 验收：回答数量增加时，单轮扫描耗时不会近似按页面全部按钮数量持续恶化；面板更新不再重复执行固定选择器查询。

- [x] **验证并移除无效或无必要的合成 wheel 事件**
  - 现状：`dispatchWheel()` 同时向 `window` 和 `document` 派发非可信事件；站点若要求 `event.isTrusted`，该事件无法模拟真实用户输入，还可能重复触发监听器。
  - 重构：通过实际验证确认知乎懒加载是否依赖该事件；若没有明确收益则删除，只保留原生滚动；若确有用途，只向必要目标派发并记录原因。
  - 验收：移除或收窄后，自动加载能力不回退，且每次滚动不会产生两次无意义 wheel 事件。

- [x] **补充纯逻辑测试与真实页面冒烟清单**
  - 纯逻辑测试至少覆盖：配置 clamp、文本规范化、点击目标匹配、空闲/到底状态转换、暂停取消、配置迁移。
  - DOM 测试至少覆盖：回答区目标会点击，评论区/侧栏/面板目标不会点击，新增节点增量处理。
  - 真实页面冒烟至少覆盖：开始、暂停、底部回弹、修改速度/间隔、拖动面板、刷新恢复、站内切换问题。
  - 验收：关键调度和点击策略可在不打开知乎页面的情况下回归；发布前冒烟步骤有明确结果记录。

## P2：体验与代码卫生

- [x] **改进控制面板可访问性和位置恢复**
  - 为 range 增加可读 label，为状态区域增加合适的 `aria-live`，为按钮明确 `type="button"`。
  - 使用 Pointer Events 支持鼠标、触控和触控笔；窗口尺寸变化或恢复保存位置时，将面板重新限制在可视区域内。

- [x] **限制调试 API 的页面暴露范围**
  - 现状：`window.zhihuAutoExpand` 可被页面脚本调用和覆盖。
  - 重构：将调试 API 设为显式可选，或通过不可枚举、只读属性暴露；在销毁时移除；若改用油猴沙箱，明确调试方式。

- [x] **清理无效状态和重复说明**
  - `lastScrollY` 当前只被赋值、未参与判断，应删除或真正用于进度判断。
  - 合并散落的魔法等待时间，明确哪些是用户设置、哪些是内部策略参数。
  - 精简只复述代码行为的长注释，保留“为什么这样做”、限制条件和容易回归的规则。

- [x] **让到底判断使用更可靠的进度信号**
  - 现状：主要根据滚动位置、页面高度和点击数判断进度；虚拟列表或高度不变的新内容可能导致误判。
  - 重构：优先记录已发现的回答 ID/回答节点数量和“加载更多”状态，滚动高度只作为辅助信号。
  - 验收：页面高度不变但新增回答时不会自动暂停；确实没有新增内容时仍能在有限轮次后暂停。
